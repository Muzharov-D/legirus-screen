## Известные gaps (не извлечено)

### G1. Дистанции по 15-мин бакетам (pages 21-35)

Каждая индивидуальная страница содержит 4 bar chart-диаграммы:
- Total distance по бакетам [0-15, 15-30, 30-45, 45-60, 60-75, 75-90]
- Distance 4 - 5.5 m/s по тем же бакетам
- Distance 5.5 - 7 m/s
- Sprint distance

**Проблема:** значения в bar chart позиционируются по координатам столбцов, и pdftotext не сохраняет порядок (например, для Турапина значения 319.0, 268.9, 248.8, 262.8, 177.8, 181.1 для 4-5.5 m/s появляются в произвольном порядке после text-extract).

**Объём gap:** 4 метрики × 6 бакетов × 15 игроков = **360 значений**.

**План:** Code должен реализовать парсер на основе `pdftotext -layout` с разбором по координатам столбцов или на основе `pdf-img-extract` + OCR. На MVP не блокирует.

**Workaround:** в Match-totals значения суммированы и доступны (`stats.fitness.totalDistance`, `speed_4_5_5`, `speed_5_5_7`, `speed_7plus`, `sprintDistance`).

### G2. Shot map / Pass map / Heat map ✓ закрыто

**Изначально:** Pages 12-20 содержат тематические карты (Карта ударов, Карта событий, Карта прессинга и т.д.) — это **векторная графика** (paths), а не raster и не текст. Числа в зонах поля недоступны через pdftotext.

**Решение:**
- На стадии build seed-данных рендерим страницы PDF через `pdftocairo -png -r 200`, кропаем регионы карт по фиксированным координатам и сохраняем в `frontend/public/assets/maps/` как PNG.
- 8 командных карт извлечены: shooting, set-pieces, passes, attacks, recoveries, duels, pressing, positioning.
- 30 индивидуальных карт извлечены: для каждого из 15 игроков — `attack-map` (карта пасов/ударов с цветными маркерами) + `fitness-heatmap` (тепловая карта движения).
- Ссылки добавлены в `match-001.json`:
  - `teamAggregates.{section}.mapImage`
  - `players[].maps.{attackMap,fitnessHeatmap}`
- При загрузке нового PDF Sportvisor через `/api/upload-pdf` Code должен запустить такой же кроп-скрипт. Скрипт сохранён как `backend/parsers/crop_maps.py` + `crop_player_maps.py`.

**Альтернатива «перерисовка SVG»:** в `frontend_starter/components/SoccerFieldZoneMap.jsx` готов React-компонент, который рисует поле + зональные счётчики + точки событий из JSON-данных. Если в будущем будет API/OCR извлечение зональных значений — карты можно полностью пересоздавать на лету.

### G3. Soccer field formation diagram (page 1)

Расстановка игроков на схеме поля сейчас восстановлена через `formation.starters[].positionSlot` (текстовая позиция типа "Центральный нападающий"). Точные **координаты** (x, y) на поле — нет.

**Объём gap:** 15 пар координат.

**План:** для FormationField компонента использовать стандартную раскладку 4-3-3 / 4-4-2 / иную, выводимую из позиций. Это паттерн UI, не данные.

### G4. Игроки соперника

Page 1 правый блок: «Нет данных об игроках команды. Пожалуйста, укажите данные...».

В PDF **отсутствуют** statы соперника по игрокам.

**План:** показать placeholder + disabled-кнопку «Назначить игроков». Это документировано как `match.guestTeamPlaceholder`.

### G5. Главный тренер

Имя тренера в PDF не указано.

**План:** поле `teams[0].headCoach = null`. UI показывает прочерк.

### G6. Логотипы клубов

В PDF логотипы есть только как PNG. Их можно извлечь через `pdfimages` (страница 1), но автоматическое сопоставление "это лого Легирус, это — соперника" — задача для Code.

**План:** для MVP пользователь загружает `frontend/public/assets/logos/legirus.png` и `porohovchanin.png` вручную. Хранилище создано.

### G7. Фото игроков

В PDF на pages 1, 21-35 есть photo headshots. Они извлекаются через `pdfimages -all`.

**Сложность:** на каждой индивидуальной странице 14 PNG, из них 8 — повторяющиеся template assets (frame, logo, иконки). 1 из оставшихся — фото игрока, остальные — графические декорации диаграммы.

**План:** для MVP пользователь загружает фото вручную в `frontend/public/assets/players/{playerId}.png`. Хранилище создано пустым.

При желании Code может реализовать heuristic: на каждой странице 21-35 извлечь все PNG, отсортировать по уникальности (хеш) и размеру; самое большое уникальное = фото. Пользователь подтверждает имя.

### G8. Категории Acceleration, Sprint sideways, Intense running как splits

В page 21-35 fitness-блоке некоторые метрики (`Sprint sideways`, `Intense running`, `Acceleration`) показываются как Match-only без разбивки 1/2 тайм. Они захвачены в `players[].stats.fitness` для match-total, но не в `splits`.

**Не gap, а структурное отличие.** Документировано.

## Верификация

```
Numeric tokens in PDF (raw extract):  9 472
Numeric leaves in match-001.json:     6 782
Splits triples (M+1+2):              1 575 троек = 4 725 значений
Total represented in JSON:           ~ 8 100 значений (с учётом split-троек)
```

Разница ~1370 значений — это:
- 360 значений из G1 (fitness buckets)
- 6×15 = 90 bucket-label токенов (`0-15`, `15-30`...) — не данные
- ~30 рейтинговых заголовков "10.0", "9.0"... на радарах (axis labels, не данные)
- ~150 повторов чисел в макете (например в page 1 формации номера и рейтинги повторяются как `17 17 17 21 21 15 15 15` из-за визуального стайлинга)
- ~750 — accuracy/success rate производные значения (в JSON хранятся как составные `{value, pct}`, в PDF появляются как 2 числа: `8 (38%)` → один value+pct в JSON, два числа в подсчёте)

**Заключение:** значимые данные перенесены без потерь, кроме G1 (fitness buckets) и G7 (photos).
