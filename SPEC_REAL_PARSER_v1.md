# SPEC_REAL_PARSER_v1 — Промышленный парсер отчётов Sportvisor

## Контекст

Текущий «парсер» — это handcrafted-данные для одного матча 2010. Кнопка «Загрузить отчёт Sportvisor» в UI на новом PDF падает с FileNotFoundError, потому что между PDF и `build_match.py` нет реального промежуточного слоя извлечения. Цель этой работы — закрыть эту дыру и превратить пайплайн `routes/upload.js → pdfParser.js → parsers/*` в end-to-end решение, работающее для любой команды клуба.

**Срок: 2 часа. Любая регрессия match-001 — критична. Тестировать обязательно после каждого этапа.**

## Жёсткие требования

1. **Без regression**: после всех изменений `match-001` (legirus-2010 vs porohovchanin-2010) должен открываться в UI идентично текущему виду. Сравнение — пиксель в пиксель: главный экран матча, подсветка лидеров, тепловые карты команды и игроков, страница каждого из 15 игроков 2010.
2. **End-to-end**: загрузка любого PDF Sportvisor через UI должна создавать корректный `match-XXX.json`, запись в `matches.json` и набор PNG-карт. Никаких костылей с захардкоженными dict'ами.
3. **Динамический ростер**: `ID_MAP` исчезает как захардкоженная константа. Маппинг «номер игрока → player.id» строится в рантайме из `players.json` по `teamId`. Игрок, которого нет в `players.json`, либо корректно создаётся «на лету» с пометкой `provisional: true`, либо парсер падает с понятным сообщением (выбрать стратегию ниже).
4. **Идемпотентность**: повторная загрузка того же PDF должна перезаписывать матч, а не создавать дубликат.
5. **Логирование**: каждая стадия пишет короткий лог `parser.log` в каталоге матча (для отладки на проде).

## Что есть на вход

- `legirus-screen` репозиторий, ветка `main`
- `backend/parsers/build_match.py` — финальный сборщик (его НЕ переписываем целиком, только адаптируем под динамический ID_MAP)
- `backend/parsers/parse_page1.py` — handcrafted dict, **переписать целиком** на pdfplumber
- `backend/parsers/crop_maps.py`, `backend/parsers/crop_player_maps.py` — кропалки PNG из PDF, **проверить работоспособность**, при необходимости адаптировать
- Эталонный PDF от 2010 (тот, по которому делался match-001): `backend/parsers/sportvisor_legirus2010_match001.pdf` — **уже на месте**
- Эталонный PDF от 2011 (новый матч, который нужно загрузить): `backend/parsers/sportvisor_legirus2011_match002.pdf` — **уже на месте**
- Ростер 2011: **уже добавлен** в `backend/data/players.json` (14 игроков с teamId=legirus-2011, позиции пустые — парсер заполнит)
- legirus-2011: **уже активирован** в `backend/data/teams.json` (active: true)

## Разведка эталонных PDF (выполнена, опираться при разработке)

PDF 2010 — 35 страниц, PDF 2011 — 32 страницы. Оба PDF v1.4. Структура **большей части страниц идентична**, но есть две критичные разницы.

**Page 1 (сводка матча) — структура РАЗНАЯ:**
- 2010: полная схема расстановки 11+4 со «жирными» (повторяющимися глифами) номерами на форме игрока, рейтинги под каждым номером, нижний блок «Запасные» с 4 игроками. Текст плотный (~1073 символа).
- 2011: блок расстановки СВЁРНУТ (тренер не назначил позиции в Sportvisor). Видна только командная статистика + блок «Запасные» с 9 номерами, но без полноценной схемы. Текст разрежённый (~474 символа). На странице Sportvisor показывает плейсхолдер «Назначить игроков».

Парсер `parse_page1.py` обязан корректно обрабатывать оба случая: если расстановка отсутствует — поле `formation` в выходном JSON ставится `null` или пустой формат, фронтенд должен это уметь.

**Pages 2+ (таблицы метрик) — структура ИДЕНТИЧНА:**
- Page 2: Overall Index — `<номер> <Фамилия/Имя> <инициал> <позиция> <минуты> <17 столбцов значений>`
- Page 3: Fitness — 9 столбцов
- Page 4: Attack 1/5 — 8 столбцов
- Дальше: Attack 2/5..5/5, Defence 1/3..3/3
- Каждая страница начинается с шапки `19.04.2026 <названия> Player Stats <раздел>`

Главное — **по номеру игрока** (первый столбец) определяется player.id через лукап в `players.json` по `(teamId, number)`. Имя/инициал в PDF — только подсказка, источник истины — players.json.

Внимание: формат имени игрока в таблицах **разный между PDF**:
- 2010: «17 Турапин М. ЦП ...» — фамилия + инициал имени
- 2011: «01 Иван З. ВР ...» — имя + инициал фамилии

Это **не блокер**, потому что мэтчим по номеру, но парсер не должен полагаться на порядок «фамилия/имя».

**Соперник 2011 — «Пороховчанин 2011»:**
- В `teams.json` сейчас есть только `porohovchanin-2010`. Для PDF 2011 парсер на этапе 3 при стратегии Б упадёт с «awayTeam not found». Перед запуском на 2011 нужно добавить `porohovchanin-2011` в `teams.json` (active: false, isOurTeam: false). Это **обязательное предусловие**, добавляется руками одной строкой.

**Жирные глифы на page 1:**
- Sportvisor рисует крупные номера на форме повторением глифа поверх себя. `extract_text()` возвращает «1111177777» вместо «17». На pages 2+ этого нет. На page 1 это нужно учесть только при чтении расстановки. Решение: использовать `page.extract_words()` с `keep_blank_chars=False`, дедупликация по координатам с tolerance 1px.

## Архитектура (строго в этом порядке)

### Этап 1. Фикстуры и golden-файлы (10 минут)

1. PDF-ки уже на месте: `backend/parsers/sportvisor_legirus2010_match001.pdf` и `backend/parsers/sportvisor_legirus2011_match002.pdf`.
2. Создать каталог `backend/parsers/golden/`.
3. Снять «золотую» копию текущего `backend/data/matches/match-001.json` в `backend/parsers/golden/match-001.golden.json`. Это эталон для regression-теста.
4. Снять копию первых 15 игроков (с teamId=legirus-2010) из `backend/data/players.json` в `backend/parsers/golden/players-2010.golden.json`.

### Этап 2. Низкоуровневый PDF-extractor (30 минут)

Создать `backend/parsers/lib/pdf_extract.py`:

- `extract_text_blocks(pdf_path)` → список `[{page, x0, y0, x1, y1, text, font, size}]`. Использовать `pdfplumber.Page.chars` агрегированно по словам.
- `extract_tables(pdf_path, page_num)` → список таблиц с явной геометрией. `pdfplumber.Page.extract_tables()` с настройками `vertical_strategy="lines"`, `horizontal_strategy="lines"`. Если sportvisor использует разделители-точки — `vertical_strategy="text"` как fallback.
- `extract_image(pdf_path, page_num, bbox)` → `PIL.Image`. Для тепловых карт. Использовать `pdfplumber.Page.crop(bbox).to_image(resolution=200)`.
- Покрыть юнит-тестом `tests/test_pdf_extract.py`: на эталонном PDF проверить, что вытаскивается ожидаемое количество страниц и хотя бы одна таблица с известным заголовком.

### Этап 3. Парсер страницы 1 — общая статистика (25 минут)

Полностью переписать `backend/parsers/parse_page1.py`:

- На вход: путь к PDF, `team_id` (наш), `match_id`.
- Найти на странице 1 блоки: дата матча, названия команд, счёт, основная статистика (владение, удары, xG, передачи, угловые, нарушения, карточки, офсайды), формация (стартовый состав 11 + замены).
- Сопоставление команд: по тексту названия. Если название содержит «Легирус» с указанным годом — это наша команда, `homeTeamId = team_id`. Иначе — гость, `awayTeamId` ищется в `teams.json` по совпадению `name`. Если гостя нет в `teams.json` — создать запись через `services/dataLoader.js` (пометить `provisional: true`) ИЛИ упасть с понятным сообщением (выбрать стратегию).
- На выходе: `page1_summary.json` рядом с PDF, со схемой как у текущего handcrafted dict (см. строки 6-60 текущего `parse_page1.py`).
- Юнит-тест: на эталонном PDF 2010 диктовать золотой `page1_summary.json` (создать вручную из match-001, как ground truth) и сравнивать с парсингом — расхождения только в плавающих числах с допуском 0.1.

### Этап 4. Парсеры таблиц команды (30 минут)

Создать `backend/parsers/parse_team_tables.py`:

- Sportvisor содержит 9 таблиц с метриками по игрокам: `overall`, `fitness`, `attack1-5`, `defence1-3`. Каждая таблица — на отдельной странице или в отдельном блоке.
- Найти таблицу по характерному заголовку (например «ОБЩИЙ ИНДЕКС», «ФИТНЕС», «АТАКА 1», и т.д.). Использовать text blocks из этапа 2 для поиска якорей, затем `extract_tables` в окрестности.
- Каждая таблица — `{ "<player_number>": { "<metric_key>": value, ... } }`. Список метрик по таблицам — взять из текущего `match-001.json` (анализ `players[i].stats.<table>`).
- Числа в русской типографике: пробел как разделитель тысяч, запятая как десятичный (`"1 234,5"` → `1234.5`). Учесть.
- На выходе: `team_tables.json`.
- Юнит-тест: для эталонного PDF 2010 ожидаем 15 номеров игроков во всех таблицах overall (где играл каждый); сравнение с golden match-001.

Создать `backend/parsers/parse_team_aggregates.py`:

- Командные агрегаты (суммарные значения по матчу для своей и чужой команды). Структура — как в `match-001.teamAggregates`.
- На выходе: `team_aggregates.json`.

### Этап 5. Парсер сплитов игроков (15 минут)

Создать `backend/parsers/parse_player_splits.py`:

- Для каждого игрока — разбиение `M / 1 тайм / 2 тайм` по ключевым метрикам, плюс `fullName`, `lastName`, `firstName`, `positionFull`. Эти данные обычно на странице игрока в Sportvisor (если в отчёте они есть; иначе скипаем `splits` и заполняем только имена).
- Имена игроков ищем по странице игрока: «<Фамилия> <Имя>» или «<Имя> <Фамилия>» — детектить.
- На выходе: `player_splits.json`.

### Этап 6. Адаптация build_match.py (10 минут)

- Убрать захардкоженный `ID_MAP`.
- Загружать `players.json`, фильтровать по `args.team_id`, строить `id_map = { str(p.number): p.id for p in players_of_team }`.
- Если в `team_tables.overall` встречается номер, которого нет в `id_map` — стратегия А: создать `provisional` игрока с `id=p<num>-unknown`, добавить в `players.json` (или временный буфер) с `teamId=args.team_id` и `provisional: true`. Стратегия Б: вернуть ошибку с понятным сообщением «Игрок №42 не найден в ростере legirus-2011 — добавьте его перед загрузкой».
  - **Выбрать стратегию Б** (промышленность требует контролируемого ростера, не тихих сюрпризов).
- Сохранить корректный teamId на выходе.

### Этап 7. Crop карт (10 минут)

- Прогнать `crop_maps.py` и `crop_player_maps.py` на эталонном PDF 2010 → сравнить с существующими PNG в `frontend/public/assets/maps/match-001/`. Если расхождения — поправить координаты bbox.
- Прогнать на PDF 2011 → проверить, что PNG создаются под `frontend/public/assets/maps/match-002/`.

### Этап 8. Адаптация upload-флоу (5 минут)

- В `backend/services/pdfParser.js` — убедиться, что промежуточные JSON-ы пишутся в каталог рядом с PDF (не в `os.tmpdir`, а в стабильный подкаталог `intermediate/match-XXX/`), чтобы их можно было посмотреть при отладке.
- В `backend/routes/upload.js` — при ошибке парсинга возвращать в `res.status(500)` структурированный объект `{ error, stage, details }`, чтобы фронтенд мог показать пользователю, на каком этапе споткнулся парсер (для team_coach это важно).

## Тестирование (regression-чек, 10 минут)

Создать `backend/parsers/tests/test_e2e.py`:

1. Запустить полный пайплайн на эталонном PDF 2010 → `match-temp.json`.
2. Сравнить с `golden/match-001.golden.json` рекурсивно, с допуском 0.1 на float-полях.
3. Любое расхождение — fail.

```bash
cd backend && python -m pytest parsers/tests -v
```

Запустить локально и в CI (Render автоматически не запустит, но в logs можно прогнать вручную через Shell).

Дополнительно — UI smoke-test:
1. Локально: `npm run dev` в frontend + `npm run dev` в backend.
2. Логин как head_coach.
3. Открыть Легирус 2010 → MatchesDashboard → match-001 → проверить, что страница идентична текущему prod (визуальное сравнение).
4. Загрузить через UI PDF от 2011 → дождаться завершения парсинга → проверить, что появилась запись в списке матчей у команды 2011, открывается дашборд, лидеры, карты.

## Что НЕ делаем в эту работу (явно out of scope)

- OCR сканированных PDF (предполагаем, что Sportvisor отдаёт текстовый PDF).
- Поддержку нестандартных шаблонов отчётов — только тот шаблон, который в эталонных PDF.
- Авто-добавление новых соперников в `teams.json` — пока стратегия Б, ошибка с просьбой добавить.
- Auto-ростер — пока стратегия Б, парсер падает на неизвестных номерах.

Если парсер успешно работает на двух эталонных PDF — задача закрыта. Расширение под другие шаблоны/команды — следующая итерация.

## Риски и митигация

| Риск | Митигация |
|------|-----------|
| Структура PDF Sportvisor отличается между сезонами/матчами | Сначала эталон 2010, затем 2011. Если 2011 ломается — запросить ещё один PDF и сравнить геометрию. |
| Кириллица в pdfplumber (некоторые шрифты) | Использовать `pdfplumber.utils.extract_text(layout=True)`. При проблемах — fallback на pymupdf. |
| Разделители тысяч/десятичные в русской типографике | Утилита `parse_ru_number(s)` в `lib/pdf_extract.py` с regex `^(\d{1,3}(?: \d{3})*)(?:,(\d+))?$`. |
| Тепловые карты — это растровые изображения, не векторы | `pdfplumber.Page.crop(...).to_image(resolution=200)` → `PIL.Image.save(png)`. |
| Игрок отсутствует в одной из таблиц (не играл) | Skip отсутствующего номера в этой таблице, не fail. |
| Regression match-001 | Golden-test обязателен, прогон ДО любого коммита. |
| Неуспеть за 2 часа | Минимальный must-have: этапы 2 + 3 + 4 (overall + одна таблица атаки + одна защиты) + 6 + 7 + golden-тест на match-001. Остальное — следующая итерация. |

## Если что-то идёт не так

- **Парсер падает на неизвестной структуре блока** — НЕ хардкодить под текущий PDF, а дописать общий extractor. Если совсем тупик — отдельный коммит с пометкой «TODO: shape X not handled» и переход к следующему этапу. Цель — иметь рабочий E2E на эталонах, а не идеальное покрытие.
- **Regression match-001** — НЕ мерджить. Ройся в golden-тесте до синхронизации.
- **2 часа закончились, но не все этапы готовы** — мерджить только если match-001 не сломан и golden-тест зелёный. Если сломан — откат и доделка вне дедлайна.
